
# CAPAS (ESTILO) Y MVC (PATRÓN) 

## **Alta cohesión y bajo acoplamiento dentro del servidor**
- El servidor no es una unidad monolítica: debe gestionar **lógica de negocio compleja** (validación de metadatos por tipo de activo, flujos de revisión, políticas de acceso).
- **MVC + servicios** permite separar:
  - **Entrada/salida (Controlador)** → maneja HTTP.
  - **Lógica de dominio (Servicio)** → implementa reglas como RF-1-2-4 (Abstract ≥100 caracteres) o DR-16 (envío obligatorio a revisión).
  - **Persistencia (Modelo)** → interactúa con la base de datos.

Esto evita que la lógica se mezcle con la comunicación HTTP o con el acceso a datos.

---


## **Separación clara de responsabilidades (mantenibilidad y evolución)**
- **Controladores**: gestionan solo la entrada/salida HTTP (req/res), sin lógica de negocio.
- **Modelos**: gestionan únicamente la persistencia.

> Esto evita que el código se vuelva monolítico y permite modificar reglas sin afectar interfaces.


---

## **Escalabilidad y preparación para evolución arquitectónica**
- La capa de servicios actúa como **interfaz estable** entre la web (controladores) y la persistencia (modelos).
- Facilita futuras migraciones

---

## **Preparación para escalabilidad y evolución**
- Soportar 10,000 usuarios concurrentes → requiere que el servidor sea modular y fácil de escalar.
- Una arquitectura en capas con **MVC + servicios** permite:
  - Desplegar componentes independientes .
  - Reemplazar la persistencia sin afectar la lógica.
  - Aislar fallos (ej: error en notificaciones no bloquea la subida de activos).
