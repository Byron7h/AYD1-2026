## **ALMACENAMIENTO Y CLASIFICACIÓN**

### CU1-1 Subir Activo (Borrador)

| Campo | Descripción |
|-------|-------------|
| **Nombre del caso de uso** | Subir Activo (Borrador) |
| **Actores** | Investigador (iniciador) |
| **Propósito** | Permitir al investigador crear un borrador de activo (paper, dataset, código fuente o documento técnico) para su posterior envío a revisión. |
| **Resumen** | El investigador inicia la carga de un nuevo activo seleccionando el tipo correspondiente. Se solicita los metadatos obligatorios según el tipo de activo y valida la estructura del archivo. Si la validación es exitosa, el activo queda almacenado en estado "Borrador". |
| **Condiciones previas** | El investigador debe estar autenticado en el sistema. |
| **Secuencia normal** | <table><tr><th>Pasos</th><th>Acción</th></tr><tr><td>1</td><td>El investigador registra un nuevo activo.</td></tr><tr><td>2</td><td>Se solicita seleccionar el tipo de activo (Paper, Dataset, Código Fuente o Documento Técnico).</td></tr><tr><td>3</td><td>El investigador selecciona el tipo y carga el archivo fuente.</td></tr><tr><td>4</td><td>El investigador completa los metadatos obligatorios según el tipo seleccionado.</td></tr><tr><td>5</td><td>Se invoca el caso de uso "Validar Metadatos de Tipo de Activo".</td></tr><tr><td>6</td><td>Se valida la información y guarda el activo en estado "Borrador".</td></tr><tr><td>7</td><td>Se confirma la creación del borrador.</td></tr></table> |
| **Excepciones / Cursos alternativos** | <table><tr><th>Pasos</th><th>Acción</th></tr><tr><td>4</td><td>Si el formato del archivo no es válido, no se permite la carga y muestra un mensaje de error especificando el formato esperado.</td></tr><tr><td>5</td><td>Si faltan campos obligatorios, se señala los campos requeridos.</td></tr><tr><td>6</td><td>Si ocurre un incoveniente en el almacenamiento, se notifica al usuario y permite reintentar.</td></tr></table> |
| **Condiciones posteriores** | El activo queda registrado en la base de datos con estado "Borrador" y asociado al investigador creador. |

### CU1-2 Validar Metadatos de Tipo de Activo

| Campo | Descripción |
|-------|-------------|
| **Nombre del caso de uso** | Validar Metadatos de Tipo de Activo |
| **Actores** | *(funcionalidad interna del sistema, no requiere actor externo)* |
| **Propósito** | Verificar que los metadatos ingresados cumplan con las especificaciones obligatorias según el tipo de activo (Paper, Dataset, Código Fuente o Documento Técnico). |
| **Resumen** | Caso de uso invocado durante la carga o modificación de un activo. Se valida que todos los campos obligatorios estén completos, que los formatos sean correctos y que los datos cumplan con las reglas de negocio establecidas para cada tipo de activo. |
| **Condiciones previas** | Debe existir un activo en proceso de registro o modificación con un tipo definido. |
| **Secuencia normal** | <table><tr><th>Pasos</th><th>Acción</th></tr><tr><td>1</td><td>Se identifica el tipo de activo (Paper, Dataset, Código o Documento) basado en la elección del usuario investigador.</td></tr><tr><td>2</td><td>Se valida los campos obligatorios específicos del tipo.</td></tr><tr><td>3</td><td>SE verifica el formato del archivo fuente.</td></tr><tr><td>4</td><td>Se valida reglas adicionales (ej: mínimo de caracteres en Abstract, mínimo de tags, formato de licencias).</td><tr><td></td><td><table><tr><td>4a</td><td>Para Documentos Técnicos: se valida que exista el campo de confidencialidad (Público/Interno).</td></tr><tr><td>4b</td><td>Para Papers: se valida Abstract mínimo 100 caracteres y mínimo 3 tags.</td></tr><tr><td>4c</td><td>Para Datasets: se ejecuta la extracción de primeras 5 filas</td></tr><tr><td>4d</td><td>Para Código Fuente: se valida que existan dependencias y stack tecnológico (Lenguaje (ej. Python) y Framework (ej. Django)).</td></tr></table></td></tr></tr><tr><td>5</td><td>El sistema confirma la validación exitosa.</td></tr></table> |
| **Excepciones / Cursos alternativos** | <table><tr><td>Pasos</td><td>Acción</td></tr><tr><td>2-4</td><td>Si alguna validación falla, se genera un listado de errores específicos indicando qué campos o reglas no se cumplen.</td></tr><tr><td>3</td><td>Si el formato de archivo es incorrecto (ej: se esperaba PDF y se cargó DOCX en un Paper), se rechaza con mensaje específico.</td></tr></table>|
| **Condiciones posteriores** | Se retorna confirmación de validación exitosa o lista de errores para corrección. |
| **Comentarios** | El caso de uso es invocado desde "Subir Activo", "Modificar Borrador" o "Enviar a Revisores" |

### CU1-3 Gestionar Borrador (Modificar/Eliminar)

| Campo | Descripción |
|-------|-------------|
| **Nombre del caso de uso** | Gestionar Borrador (Modificar/Eliminar) |
| **Actores** | Investigador (iniciador) |
| **Propósito** | Permitir al investigador editar o eliminar completamente un activo mientras se encuentre en estado "Borrador". |
| **Resumen** | El investigador revisa uno de sus borradores y decide actualizarlo o retirarlo (metadatos o archivo). Si modifica, se valida nuevamente los metadatos antes de guardar. Si elimina, se solicita confirmación antes de realizar la eliminación permanente. |
| **Condiciones previas** | El activo debe existir en estado "Borrador". El investigador debe ser el propietario del activo. El activo no debe haber sido enviado a revisión. |
| **Secuencia normal** | <table><tr><th>Pasos</th><th>Acción</th></tr><tr><td>1</td><td>El investigador selecciona un borrador de su lista de activos.</td></tr><tr><td>2</td><td>Se muestra los datos actuales del activo (incluido su estado) con opciones de "actualizar" o "retirar".</td></tr><tr><td></td><td><table><tr><td>2a</td><td>**Flujo Modificar:** El investigador selecciona "Modificar" y edita los campos deseados (metadatos o archivo).</td></tr><tr><td>2a.1</td><td>El investigador confirma los cambios.</td></tr><tr><td>2a.2</td><td>Se "Validar Metadatos de Tipo de Activo".</td></tr><tr><td>2a.3</td><td>Se registra en la bitacora la modificación y la fecha en que se hizo.</td></tr><tr><td>2a.4</td><td>Se confirma la actualización exitosa.</td></tr></table></td></tr><tr><td></td><td><table><tr><td>2b</td><td>**Flujo Eliminar:** El investigador selecciona "Eliminar Borrador".</td></tr><tr><td>2b.1</td><td>Se solicita confirmación mostrando advertencia: "Esta acción eliminará permanentemente el borrador. ¿Desea continuar?".</td></tr><tr><td>2b.2</td><td>El investigador confirma la eliminación.</td></tr><tr><td>2b.3</td><td>Se retira el borrador de la base de datos y todos sus archivos asociados.</td></tr><tr><td>2b.4</td><td>Se confirma la eliminación exitosa y retorna a la lista de borradores.</td></tr></table></td></tr></table> |
| **Excepciones / Cursos alternativos** | <table><tr><th>Pasos</th><th>Acción</th></tr><tr><td>1</td><td>Si el activo no está en estado "Borrador" el sistema niega el acceso.</td></tr><tr><td>2a.3</td><td>Si la validación falla en modificación, se muestran los errores y no se guardan los cambios hasta corregirlos.</td></tr><tr><td>2a.4</td><td>Si hay error de almacenamiento en modificación, se notifica y permite reintentar.</td></tr><tr><td>2b.2</td><td>Si el investigador cancela la confirmación de eliminación, el sistema retorna a la vista del borrador sin cambios.</td></tr><tr></table> |
| **Condiciones posteriores** | **Si modificó:** El borrador queda actualizado con la nueva información y mantiene el estado "Borrador". **Si eliminó:** El borrador es removido permanentemente del sistema. |

### CU1-4 Enviar Activo a Revisores

| Campo | Descripción |
|-------|-------------|
| **Nombre del caso de uso** | Enviar a Revisores |
| **Actores** | Investigador (iniciador) |
| **Propósito** | Enviar un activo en estado "Borrador" al proceso de revisión para su aprobación antes de publicación. |
| **Resumen** | El investigador selecciona un borrador completo y solicita enviarlo a revisión. Se valida que todos los metadatos estén correctos, cambia el estado del activo a "En Revisión" y notifica a los revisores disponibles. |
| **Condiciones previas** | El activo debe estar en estado "Borrador". El investigador debe ser el propietario del activo. Todos los metadatos obligatorios deben estar completos. |
| **Secuencia normal** | <table><tr><th>Pasos</th><th>Acción</th></tr><tr><td>1</td><td>El investigador selecciona un borrador de su lista.</td></tr><tr><td>2</td><td>El investigador solicita enviar a revisión.</td></tr><tr><td>3</td><td>Se registra el activo como pendiente de revisión.</td></tr><tr><td>4</td><td>Se notifica a los revisores sobre el nuevo activo pendiente.</td></tr><tr><td>5</td><td>Se confirma al investigador que el activo fue enviado a revisión.</td></tr></table> |
| **Excepciones / Cursos alternativos** | <table><tr><th>Pasos</th><th>Acción</th></tr><tr><td>4</td><td>Si falla el envío de notificaciones, se registra el error pero mantiene el cambio de estado.</td></tr></table> |
| **Condiciones posteriores** | El activo cambia a estado "En Revisión" y queda bloqueado para edición y eliminación por el investigador. Los revisores reciben notificación del activo pendiente. |

<br>

### CU1-5 Revisión/Aprobación de Activo

| Campo | Descripción |
|-------|-------------|
| **Nombre del caso de uso** | Revisión de Activo |
| **Actores** | Revisor (iniciador) |
| **Propósito** | Evaluar un activo en estado "En Revisión" y decidir si se aprueba para publicación o se rechaza con feedback. |
| **Resumen** | Un revisor examina el contenido del activo y decide si cumple con los criterios académicos, técnicos y éticos establecidos (lo cual publica e indexa automáticamente el activo) o rechazar (devolviendo al investigador con comentarios obligatorios). |
| **Condiciones previas** | El activo debe estar en estado "En Revisión". |
| **Secuencia normal** | <table><tr><th>Pasos</th><th>Acción</th></tr><tr><td>1</td><td>El revisor consulta los activos que tiene pendientes de evaluación.</td></tr><tr><td>2</td><td>El revisor selecciona un activo para revisar.</td></tr><tr><td>3</td><td>Se muestra el contenido completo del activo y sus metadatos.</td></tr><tr><td>4</td><td>El revisor decide aprobar el activo.</td></tr><tr><td>5</td><td>Se cambia el estado a "Publicado" e indexa el activo para búsqueda.</td></tr><tr><td>6</td><td>Se invoca "Registrar Cambio" con los detalles de la aprobación.</td></tr><tr><td>7</td><td>Se notifica al investigador sobre la aprobación.</td></tr></table> |
| **Excepciones / Cursos alternativos** | <table><tr><th>Pasos</th><th>Acción</th></tr><tr><td>4</td><td>Si el revisor decide rechazar, se extiende al caso de uso "Rechazar Activo".</td></tr><tr><td>5</td><td>Si hay error en la indexación, se le muestra un mensaje de error al revisor donde se le indica que lo intente de nuevo </td></tr></table> |
| **Condiciones posteriores** | Si se aprueba: el activo cambia a estado "Publicado", queda indexado para búsqueda y visible según su nivel de confidencialidad (público/privado). El investigador recibe notificación de aprobación. |

### CU1-6 Rechazar Activo

| Campo | Descripción |
|-------|-------------|
| **Nombre del caso de uso** | Rechazar Activo |
| **Actores** | Revisor (iniciador) |
| **Propósito** | Rechazar un activo en revisión debido a problemas de calidad, plagio o citas incorrectas, devolviendo el activo al investigador con feedback obligatorio. |
| **Resumen** | El revisor, durante el proceso de revisión, identifica problemas que impiden la publicación (como plagio o citas mal formateadas).Se solicita feedback obligatorio, cambia el estado del activo a "Rechazado" y notifica al investigador con los comentarios de rechazo. |
| **Condiciones previas** | El activo debe estar en estado "En Revisión". El revisor debe estar en el proceso de revisión del activo. |
| **Secuencia normal** | <table><tr><th>Pasos</th><th>Acción</th></tr><tr><td>1</td><td>Dentro del caso de uso "Revisión de Activo", el revisor decide rechazar.</td></tr><tr><td>2</td><td>Se solicita al revisor ingresar feedback obligatorio explicando el motivo del rechazo (ej: plagio detectado, citas mal formateadas).</td></tr><tr><td>3</td><td>El revisor ingresa los comentarios detallados del feedback.</td></tr><tr><td>4</td><td>SE cambia el estado del activo a "Borrador".</td></tr><tr><td>5</td><td>Se registra el cambio invocando "Registrar Cambio" con los detalles del rechazo.</td></tr><tr><td>6</td><td>Se notifica al investigador con los comentarios del revisor.</td></tr></table> |
| **Excepciones / Cursos alternativos** | <table><tr><th>Pasos</th><th>Acción</th></tr><tr><td>2</td><td>Si el revisor no ingresa feedback, el sistema no permite completar el rechazo.</td></tr></table> |
| **Condiciones posteriores** | El activo vuelve a estar disponible para modificación, para que el investigador pueda corregirlo. El investigador recibe notificación con el feedback del revisor. |

<br>


### CU1-7 Registrar Cambio

|-------|-------------|
| **Nombre del caso de uso** | Registrar Cambio |
| **Actores** | *(funcionalidad interna del sistema, no requiere actor externo)* |
| **Propósito** | Mantener una bitácora completa de todos los cambios de estado y modificaciones realizadas a los activos del sistema para auditoría y trazabilidad. |
| **Resumen** | Caso de uso invocado automáticamente cada vez que ocurre un cambio significativo en un activo (aprobación, rechazo, retiro de publicación, modificación). Registra quién realizó el cambio, cuándo, qué tipo de cambio fue y cualquier comentario asociado. |
| **Condiciones previas** | Debe existir un activo en el sistema. Debe haberse ejecutado una acción que modifique el estado o contenido del activo. |

| **Excepciones / Cursos alternativos** | <table><tr><th>Pasos</th><th>Acción</th></tr><tr><td>7</td><td>Si se encuentra un inconveniente al registrar el cambio, se notifica error donde se le indica que lo intente de nuevo</td></tr></table> |
| **Condiciones posteriores** | Queda registrado en la base de datos un nuevo registro de auditoría con todos los detalles del cambio realizado. |
